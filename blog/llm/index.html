The Missing Foundations of LLM Pretraining

June 25, 2026 (1d ago)

Introduction

Large language models are often presented as a sequence of architectural innovations: Transformers, attention mechanisms, scaling laws, and post-training techniques. While this perspective is useful descriptively, it hides a more fundamental structure that explains why these systems work, how they scale, and—most importantly—why they fail.

This document is an attempt to reconstruct that structure from first principles.

Rather than treating pretraining as a collection of disconnected techniques, we view it as a single coupled system governed by a small number of constraints:

  • Compute, which limits how much learning can occur
  • Data, which defines what is learned
  • Optimization dynamics, which determine how learning unfolds
  • Memory and state, which constrain what can be stored and reused
  • System design, which determines what is computationally feasible

From this perspective, many well-known ideas in modern machine learning become different expressions of the same underlying tradeoffs. Scaling laws describe compute allocation. Data curation defines the training distribution. Attention is a memory system under bandwidth constraints. FlashAttention, checkpointing, KV caching, and distributed training are all different ways of managing state under hardware limits.

The goal of this document is not to provide a catalog of techniques, but to develop a unified mental model that allows one to reason about modern LLM training and inference systems in a principled way. In particular, it focuses on answering a question that frequently appears in frontier lab interviews:

When a large language model system fails or scales poorly, what is the underlying constraint that was violated?

By the end, most pretraining and inference behaviors can be interpreted as consequences of a small set of interacting design principles rather than as isolated engineering tricks.

Scaling Laws: Compute Allocation, Not Just Bigger Models

A common simplification when discussing large language models is:

Bigger models are better.

This statement is partially true, but it misses the most important idea behind modern LLM scaling.

The real question is not:

How large should the model be?

The real question is:

Given a fixed amount of compute, how should we allocate resources between model capacity and training experience?

This shift in perspective is the foundation of modern scaling laws.

The two fundamental resources of pretraining

A language model is primarily controlled by two quantities:

  • the number of parameters
  • the amount of training data

These two resources compete for the same computational budget.

Model parameters: increasing model capacity

Let:

N=number of parametersN = \text{number of parameters}N=number of parameters

Increasing NNN gives the model more capacity:

  • more representational power
  • more ability to store patterns
  • ability to learn more complex functions

A larger model can represent a richer set of solutions.

However, capacity alone is not sufficient.

A model with billions of parameters that does not receive enough training examples is under-utilized.

Training tokens: increasing model experience

Let:

D=number of training tokensD = \text{number of training tokens}D=number of training tokens

Increasing DDD gives the model more experience:

  • more examples of language
  • better estimation of the data distribution
  • improved generalization

The model does not learn from parameters alone.

It learns from the interaction between parameters and data.

Compute is the constraint

Training compute is approximately:

C≈N×DC \approx N \times DC≈N×D

where:

  • NNN is the number of parameters
  • DDD is the number of training tokens

This means increasing model size reduces the amount of data that can be processed under a fixed compute budget.

The question becomes:

Should additional compute buy a larger model or more training tokens?

The early intuition: bigger is always better

Early scaling experiments showed a clear relationship:

  • larger models performed better
  • more parameters improved benchmark results

This created a simple intuition:

more parameters→better model\text{more parameters} \rightarrow \text{better model}more parameters→better model

Therefore, when additional compute became available, the dominant strategy was:

Build a larger model.

However, this ignored whether the model was trained enough.

The scaling law work showed that language model loss follows predictable power-law behavior.

A simplified relationship is:

  • LLL is the loss
  • NNN is the number of parameters
  • L∞L_\inftyL∞​ is the irreducible loss

The important observation:

Increasing model size improves performance in a predictable way.

However, the same phenomenon exists for data:

L(D)L(D)L(D)

More tokens also improve performance.

Therefore the real optimization problem is:

min⁡L(N,D)\min L(N,D)minL(N,D)

under the compute constraint:

C=N×DC=N\times DC=N×D

The hidden problem: under-trained models

A very large model can fail to reach its potential if it does not see enough data.

A model has two sources of error:

L=Lcapacity+LdataL = L_{\text{capacity}} + L_{\text{data}}L=Lcapacity​+Ldata​

A small model suffers from limited capacity:

Lcapacity↑L_{\text{capacity}} \uparrowLcapacity​↑

Ldata↑L_{\text{data}} \uparrowLdata​↑

The second case explains why a smaller model trained longer can outperform a larger model.

Chinchilla: compute-optimal training

The Chinchilla scaling laws revisited the question:

Given a fixed compute budget, what is the optimal allocation between parameters and data?

The key conclusion:

Many large language models were under-trained.

They had:

  • too many parameters
  • too few training tokens

A simplified compute-optimal rule is:

D≈20ND \approx 20ND≈20N

meaning:

For each parameter, the model should see roughly twenty training tokens.

For example:

A 70B parameter model would ideally require approximately:

Why smaller models can outperform larger models

A larger model is not automatically better.

A model with more parameters but insufficient training data wastes capacity.

A smaller model trained with enough tokens can learn a better approximation of the data distribution.

The important quantity is not:

number of parameters\text{number of parameters}number of parameters

but:

capacity×experience\text{capacity} \times \text{experience}capacity×experience

The deeper insight

Scaling laws changed the philosophy of LLM training.

The old mindset:

Build the biggest neural network possible.

The modern mindset:

Allocate finite compute between model capacity and training experience.

A frontier model is not defined only by its size.

It is defined by how efficiently compute is converted into useful learning.

Data Is the Hidden Model: The Training Distribution Defines Capabilities

A model does not learn from “the internet”.

It learns from the specific distribution of tokens selected during training.

The data pipeline is therefore not just preprocessing.

It is part of the model design.

The deeper insight

Scaling laws tell us:

More compute produces predictable improvements.

Data determines:

A model is not only:

The dataset is not an input to the model.

The dataset is part of the model.

Optimization Is a Stochastic Dynamical System

Training a neural network is often described as “minimizing a loss function”.

This description is correct, but incomplete.

In practice, training is not deterministic optimization. It is a noisy, discrete-time dynamical system driven by stochastic gradients, numerical stability constraints, and hardware-induced approximations.

Understanding this viewpoint is essential for reasoning about:

  • why training is unstable at scale
  • why learning rate schedules matter
  • why batch size changes generalization
  • why large models sometimes diverge unexpectedly

Training as gradient flow with noise

At a high level, gradient descent updates parameters as:

θt+1=θt−η∇θL(θt)\theta_{t+1} = \theta_t - \eta \nabla_\theta L(\theta_t)θt+1​=θt​−η∇θ​L(θt​)

In deep learning, we never compute the full gradient over the dataset.

Instead, we use stochastic estimates:

gt≈∇θL(θt)g_t \approx \nabla_\theta L(\theta_t)gt​≈∇θ​L(θt​)

so the update becomes:

θt+1=θt−ηgt\theta_{t+1} = \theta_t - \eta g_tθt+1​=θt​−ηgt​

This introduces noise:

gt=∇L(θt)+ϵtg_t = \nabla L(\theta_t) + \epsilon_tgt​=∇L(θt​)+ϵt​

where ϵt\epsilon_tϵt​ depends on batch sampling.

Thus training is not deterministic descent, but:

a stochastic process over a high-dimensional parameter space.

SGD as a discretized stochastic differential equation

In the continuous limit, SGD behaves like a stochastic differential equation:

dθ=−∇L(θ) dt+Σ(θ) dWtd\theta = -\nabla L(\theta)\,dt + \Sigma(\theta)\,dW_tdθ=−∇L(θ)dt+Σ(θ)dWt​

where:

  • −∇L(θ)-\nabla L(\theta)−∇L(θ) is the drift term (optimization signal)
  • Σ(θ)\Sigma(\theta)Σ(θ) is the noise covariance
  • dWtdW_tdWt​ is Brownian motion

This framing is important because it explains:

  • why smaller batch sizes introduce regularization
  • why large batch training can generalize worse
  • why optimization is not purely about reaching minima

We are not converging to a point—we are sampling a trajectory.

The role of learning rate as system dynamics control

The learning rate η\etaη controls the stability of the system.

  • too small → slow convergence, trapped dynamics
  • too large → divergence or chaotic oscillations

At scale, learning rate becomes a stability parameter, not just a speed parameter.

Warmup schedules exist because early training is particularly sensitive:

  • weights are uncalibrated
  • activations have high variance
  • gradients are unstable

Warmup gradually transitions the system into a stable regime.

Batch size and gradient noise scale

The batch size controls the variance of gradient estimates:

  • small batch → high noise
  • large batch → low noise

This affects the implicit bias of optimization:

  • high noise → flatter minima, better generalization (often)
  • low noise → sharper minima, faster convergence but potentially worse generalization

There is a fundamental tradeoff between:

optimization efficiencyvsstatistical regularization\text{optimization efficiency} \quad \text{vs} \quad \text{statistical regularization}optimization efficiencyvsstatistical regularization

Modern optimizers modify the raw gradient dynamics.

Adam introduces exponential moving averages:

θt+1=θt−ηmtvt+ϵ\theta_{t+1} = \theta_t - \eta \frac{m_t}{\sqrt{v_t} + \epsilon}θt+1​=θt​−ηvt​​+ϵmt​​

This introduces:

  • adaptive step sizes per parameter
  • smoothing of stochastic noise
  • implicit conditioning of the loss landscape

Adam can be interpreted as:

a preconditioned stochastic dynamical system.

Weight decay and effective regularization

With AdamW-style weight decay:

θ←θ−η(∇L(θ)+λθ)\theta \leftarrow \theta - \eta (\nabla L(\theta) + \lambda \theta)θ←θ−η(∇L(θ)+λθ)

This adds a contraction force toward zero, preventing parameter explosion.

In large models, weight decay is not optional—it stabilizes the dynamics.

Why training instability emerges at scale

As models grow, several factors interact:

  • larger gradients in early layers
  • mixed precision numerical errors
  • attention softmax saturation
  • residual stream amplification

These effects accumulate, producing:

  • loss spikes
  • divergence events
  • NaNs in activations

Instability is not a bug. It is a property of scaling nonlinear dynamical systems.

The key mental model

Training is not:

minimizing a function

but rather:

evolving a high-dimensional stochastic system under constraints of noise, curvature, and numerical precision.

The goal is not to “reach the minimum”.

The goal is to:

keep the system in a stable, information-rich regime for long enough to extract structure from data.

Attention Is a Memory Mechanism, Not a Layer

Attention is usually introduced as a neural network “layer” operating on sequences.

This is misleading.

A more accurate interpretation is:

Attention is a differentiable memory retrieval system operating over a set of key-value pairs.

Once this is understood, many engineering and scaling phenomena become natural consequences rather than separate tricks.

1. From sequence processing to memory access

Given an input sequence of hidden states:

x1,x2,…,xTx_1, x_2, \dots, x_Tx1​,x2​,…,xT​

we compute:

  • queries QQQ
  • keys KKK
  • values VVV
Q=XWQ,K=XWK,V=XWVQ = XW_Q, \quad K = XW_K, \quad V = XW_VQ=XWQ​,K=XWK​,V=XWV​

Attention is defined as:

Attn(Q,K,V)=softmax(QKTd)V\text{Attn}(Q,K,V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d}}\right)VAttn(Q,K,V)=softmax(d​QKT​)V

But this formulation hides the key idea:

Each token is querying a memory bank of past representations.

2. Keys and values define a memory

We can reinterpret the computation as:

  • Keys = addressing mechanism
  • Values = stored content

Each token writes:

(kt,vt)(k_t, v_t)(kt​,vt​)

and future tokens read from this store using similarity between queries and keys.

This makes attention equivalent to:

content-addressable memory.

3. Why attention is not “just a layer”

Unlike standard MLP layers:

  • attention has access to all previous tokens
  • computation depends on pairwise interactions
  • state grows with sequence length

Thus attention is fundamentally different:

  • MLP: local transformation
  • attention: global memory lookup

This difference is structural, not cosmetic.

4. The quadratic cost comes from memory access patterns

The core computation:

QKTQK^TQKT

computes all pairwise similarities:

  • every query compares to every key
  • resulting in T2T^2T2 interactions

This is not an implementation detail.

It is the direct consequence of:

full retrieval over an uncompressed memory.

5. Causal attention as online memory construction

In autoregressive models, we enforce:

token t can only attend to ≤t\text{token } t \text{ can only attend to } \leq ttoken t can only attend to ≤t

This transforms attention into an online system:

  • at step ttt, memory contains ttt entries
  • new tokens extend memory
  • past computations remain valid

This is what enables KV caching during inference.

6. KV cache as persistent external memory

During inference, recomputing K,VK,VK,V is redundant.

We store:

K1:t,V1:tK_{1:t}, \quad V_{1:t}K1:t​,V1:t​

This turns attention into:

a read-only memory system with append-only writes.

The cost shifts from recomputation to storage bandwidth.

This is why inference becomes memory-bound rather than compute-bound.

7. Attention as retrieval, not transformation

It is useful to separate two roles:

  • MLP layers: transform representations
  • attention layers: retrieve relevant past information

Thus a transformer block is:

computation + memory access

not simply a sequence of layers.

8. Why scaling attention increases capability

Increasing model depth or width improves:

  • representation of queries (better retrieval)
  • representation of keys (better indexing)
  • representation of values (richer stored content)

Thus scaling does not just increase “capacity”.

It improves:

the quality of memory addressing.

9. The fundamental bottleneck

Attention is limited by:

  • memory size (KV cache growth)
  • quadratic retrieval cost
  • bandwidth constraints during inference

This is why many modern systems optimizations exist:

  • FlashAttention → reduce memory traffic
  • KV caching → avoid recomputation
  • sparse attention → reduce retrieval scope

All of these are modifications of the same system:

structured memory access under hardware constraints.

10. Key mental model

A transformer is not a stack of layers.

It is:

a system that writes to memory (K,V) and repeatedly queries it (QKᵀ) under compute and bandwidth constraints.

From this perspective:

  • attention = memory retrieval
  • KV cache = persistent state
  • FlashAttention = optimized memory traversal
  • long-context modeling = memory scaling problem

Once attention is understood as a memory system, the next natural question is:

All Systems Tricks Reduce to State Management

Modern large language model systems are often described through a collection of seemingly unrelated techniques:

  • activation checkpointing
  • FlashAttention
  • KV caching
  • distributed training (FSDP / tensor parallelism)
  • MoE routing
  • gradient checkpointing
  • recomputation strategies

At first glance, these appear to be independent engineering optimizations.

However, they all optimize the same underlying constraint:

how to manage state under limited memory, compute, and communication bandwidth.

Once this is understood, most systems design questions reduce to a single unifying principle.

1. The fundamental constraint: state cannot all be resident

A transformer training step involves multiple forms of state:

  • activations (intermediate forward pass tensors)
  • parameters (model weights)
  • gradients (backward pass signals)
  • optimizer state (Adam moments, etc.)
  • attention memory (keys and values)

The key issue:

not all of this state can be stored at full precision and full granularity on a single device.

Therefore, every system design choice is a tradeoff between:

  • recomputation
  • storage
  • communication

2. Activation checkpointing = recomputation over storage

During backpropagation, intermediate activations are required to compute gradients:

∂L∂Wi\frac{\partial L}{\partial W_i}∂Wi​∂L​

Naively, all intermediate states are stored.

Checkpointing instead stores only selected intermediate states and recomputes the rest during backward pass.

This trades:

  • memory usage ↓
  • compute usage ↑

Formally:

checkpointing replaces storage complexity with recomputation complexity.

3. KV cache = temporal state persistence

During inference, attention requires past keys and values:

K1:t,V1:tK_{1:t}, \quad V_{1:t}K1:t​,V1:t​

Recomputing them at every step would be redundant.

Instead, we persist them across time.

This introduces a different tradeoff:

  • memory usage ↑ (grows with sequence length)
  • compute usage ↓ (no recomputation)

KV cache is therefore:

a temporal memory optimization for autoregressive state reuse.

4. FlashAttention = structured elimination of materialized state

Standard attention explicitly constructs:

QKTQK^TQKT

which is a full T×TT \times TT×T matrix.

FlashAttention avoids materializing this matrix by computing attention in blocks:

  • streaming over KKK and VVV
  • computing softmax incrementally
  • never storing full attention scores

This changes the bottleneck:

  • memory bandwidth ↓ dramatically
  • compute roughly unchanged
  • numerical stability improved

Thus FlashAttention is:

checkpointing applied inside the attention computation graph.

5. FSDP = distributed state sharding

Fully Sharded Data Parallel (FSDP) distributes model parameters across devices.

Instead of storing full parameters on each GPU:

  • parameters are partitioned (sharded)
  • gathered only when needed
  • gradients are reduced and redistributed

This introduces a spatial state tradeoff:

  • memory per device ↓
  • communication cost ↑

FSDP is:

checkpointing of parameters across devices.

6. MoE = conditional state activation

Mixture-of-Experts models store multiple parameter sets:

  • only a subset is activated per token
  • routing selects active experts

This changes computation pattern:

  • total parameter count ↑
  • active computation per token ≈ constant

MoE is therefore:

sparse activation of a large distributed state space.

7. The unifying principle

All these methods manipulate the same abstraction:

State = anything required to compute forward/backward passes

We can classify optimizations along three axes:

StrategyWhat is reduced
Checkpointingstored activations
KV cacherecomputation over time
FlashAttentionmemory materialization
FSDPparameter residency
MoEactive parameter subset

Despite different implementations, all techniques trade:

compute↔memory↔communication\text{compute} \leftrightarrow \text{memory} \leftrightarrow \text{communication}compute↔memory↔communication

8. The deeper insight

The key realization is:

A large language model is not just a function. It is a distributed state machine.

Training and inference are both:

  • state creation
  • state transformation
  • state movement across hardware boundaries

All engineering decisions reduce to:

where does the state live, and when do we pay the cost to access it?

9. Why this perspective matters in interviews

This unifying view is what interviewers are often probing indirectly when they ask:

  • “Why does checkpointing work?”
  • “What is KV cache really doing?”
  • “Why does FlashAttention reduce memory usage?”
  • “What breaks when you scale sequence length?”

They are not testing isolated knowledge.

They are testing whether you understand:

state, and how it moves through a system.

10. Key mental model

A transformer system is:

a distributed state management engine operating under strict memory, compute, and bandwidth constraints.

Every optimization is simply a different way of moving or avoiding state.

Pretraining Failure Modes Are Predictable

One of the most important shifts in thinking about large language model training is the realization that:

failures at scale are not random—they are structured consequences of compute, data, and optimization constraints.

Once scaling laws, data distribution, optimization dynamics, and system constraints are understood, many training failures can be anticipated rather than discovered empirically.

1. The three fundamental sources of failure

Almost all pretraining issues originate from one of three domains:

  • Compute allocation failure (scaling mismatch)
  • Data distribution failure (bad or misaligned signal)
  • Optimization failure (unstable dynamics)

These interact, but are conceptually separable.

2. Compute allocation failure (under/over-scaling)

From scaling laws, performance depends on both model size and training tokens:

L=f(N,D)L = f(N, D)L=f(N,D)

subject to:

C=N×DC = N \times DC=N×D

Under-trained models

If NNN is too large relative to DDD:

  • the model has capacity it cannot use
  • representations remain poorly fitted
  • validation loss plateaus early

This is the Chinchilla regime violation.

Over-constrained models

If NNN is too small:

  • capacity saturates early
  • additional data yields diminishing returns
  • model cannot represent long-tail structure

Key insight

Many “training failures” are actually incorrect compute allocation choices.

3. Data distribution failure

The model is trained on:

Ex∼ptrain\mathbb{E}_{x \sim p_{\text{train}}}Ex∼ptrain​​

If ptrainp_{\text{train}}ptrain​ is misaligned, the model learns the wrong inductive biases.

(a) Low-quality data contamination

If noisy or low-information tokens dominate:

  • gradients become high variance
  • optimization signal weakens
  • model learns spurious correlations

This often manifests as:

“model feels dumb despite scaling”

(b) Distribution shift

If training data differs from target use cases:

  • benchmark performance does not reflect real performance
  • reasoning ability may not transfer

(c) Duplication amplification

Repeated samples distort effective distribution:

p(x)→p(x)+δ⋅duplicatesp(x) \rightarrow p(x) + \delta \cdot \text{duplicates}p(x)→p(x)+δ⋅duplicates

The optimizer interprets frequency as importance.

4. Optimization failure modes (dynamics instability)

Training is a stochastic dynamical system:

dθ=−∇L(θ) dt+Σ(θ) dWtd\theta = -\nabla L(\theta)\,dt + \Sigma(\theta)\,dW_tdθ=−∇L(θ)dt+Σ(θ)dWt​

This system can become unstable.

(a) Loss spikes and divergence

Causes include:

  • learning rate too high
  • poor warmup schedule
  • mixed precision overflow
  • unstable attention logits

At scale, small numerical instabilities amplify across layers.

(b) Gradient explosion in deep networks

Residual streams accumulate signal:

xl+1=xl+f(xl)x_{l+1} = x_l + f(x_l)xl+1​=xl​+f(xl​)

Without proper normalization, signal magnitude grows with depth.

This leads to:

  • NaNs
  • irrecoverable divergence

(c) Batch size / noise mismatch

Too little noise:

  • sharp minima
  • poor generalization

Too much noise:

  • slow convergence
  • unstable training dynamics

5. Attention-specific failure modes

Attention introduces its own class of instabilities:

  • softmax saturation (peaky distributions)
  • attention collapse (single-token dominance)
  • KV cache drift (long-context degradation)

At long sequence lengths:

QKTd\frac{QK^T}{\sqrt{d}}d​QKT​

becomes numerically sensitive.

6. Systems-level failure modes (state explosion)

Modern training is constrained by state management:

  • activations
  • gradients
  • optimizer states
  • attention caches

Failures occur when state becomes unmanageable:

(a) Memory exhaustion

  • activations exceed device capacity
  • optimizer state dominates memory footprint

(b) Communication bottlenecks

  • distributed training becomes bandwidth-limited
  • FSDP synchronization overhead dominates compute

(c) Recompute/communication mismatch

Checkpointing improves memory usage but increases compute:

memory↓⇒compute↑\text{memory} \downarrow \quad \Rightarrow \quad \text{compute} \uparrowmemory↓⇒compute↑

Incorrect tradeoffs reduce throughput instead of improving it.

7. Emergent failure: misalignment between training signals and evaluation

Even when training is stable, failure can appear at evaluation time:

  • perplexity improves but reasoning does not
  • loss decreases but capabilities plateau
  • scaling produces no observable gains

This usually indicates:

mismatch between training objective and desired capability

not optimization failure.

8. The unifying view of failure

All pretraining failures can be mapped to a single equation:

Failure=f(compute allocation,data distribution,optimization stability,system constraints)\text{Failure} = f( \text{compute allocation}, \text{data distribution}, \text{optimization stability}, \text{system constraints} )Failure=f(compute allocation,data distribution,optimization stability,system constraints)

When something breaks, it is almost always because one of these assumptions was violated.

9. Key mental model

Pretraining is not an exploratory process.

It is a constrained dynamical system where:

  • compute defines the capacity budget
  • data defines the learning signal
  • optimization defines trajectory stability
  • systems define feasibility of execution

Failures are therefore not surprises.

They are predictable constraint violations.

Final synthesis

A frontier model is not just a neural network.

It is:

a large-scale coupled system of optimization, statistics, and distributed computation.

Understanding pretraining means understanding how that system fails under stress.
